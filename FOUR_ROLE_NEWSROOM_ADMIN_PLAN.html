Four-Role Newsroom Admin Plan

A locked newsroom operating model for Lokswami that separates business control, newsroom operations, reporting, and copy-editing into four clear admin roles.

Roles Super Admin, Admin, Reporter, Copy Editor
Core Rule Only Super Admin can access System Settings
Operations Owner Admin owns full newsroom user management

Immediate Focus

  • Reduce admin roles to the final four-role newsroom model.
  • Move day-to-day staff management into the Admin panel.
  • Keep business, revenue, and system controls with Super Admin only.

Summary

Lock the admin app to four newsroom roles only: super_admin, admin, reporter, and copy_editor.

  • Super Admin: business and platform control only
  • Admin: full newsroom operations and user management
  • Reporter: create and submit content
  • Copy Editor: quality and editing desk

Hard rule: only Super Admin can access System Settings. Admin gets full newsroom user management, but cannot create, edit, or remove a Super Admin.

Final Role Model

Role Main Purpose Allowed Panels
super_admin Owner, business, and platform control Analytics Dashboard, Revenue & Ads Control, System Settings, Audit Log, Permission Review, Operations Diagnostics
admin Newsroom operations boss User Management, Story Approval, Assignment Desk, Content Queue, Push Alerts, Newsroom Settings
reporter Submission desk Submit Stories, Upload Media, Location Tagging, Source Info, My Drafts, My Submissions
copy_editor Quality desk Edit & Proofread, Fact Checking, Headline Writing, Image Optimization, Return for Changes

User-Management Boundary

  • super_admin can create and manage all roles, including super_admin
  • admin can create and manage only admin, reporter, and copy_editor
  • admin cannot create or edit super_admin, demote the last super_admin, access System Settings, or access Revenue & Ads Control

Newsroom Workflow

Workflow States

  • draft
  • submitted
  • assigned
  • copy_edit
  • changes_requested
  • ready_for_approval
  • approved
  • scheduled
  • published
  • rejected
  • archived

Role Actions

Reporter: create draft, upload media, fill location tagging, fill source info, submit, revise on changes requested.

Copy Editor: proofread, fact check, rewrite headline, optimize image, return for changes, mark ready for approval.

Admin: review, assign, manage queue, approve/reject, schedule/publish, send push alerts, manage newsroom users.

Super Admin: analytics, ads/revenue, system state, audits, permissions, emergency override.

Workflow Path

  1. super_admin creates the first newsroom staff
  2. admin manages day-to-day newsroom users
  3. reporter creates content and submits it
  4. admin receives it in Content Queue
  5. admin assigns it to copy_editor or handles directly
  6. copy_editor either returns changes_requested or marks ready_for_approval
  7. admin approves, schedules, and publishes
  8. admin optionally sends push alerts
  9. super_admin monitors via analytics, audit, and system panels

Implementation Changes

1. Role and Permission Contract

  • Admin surfaces should only recognize super_admin, admin, reporter, and copy_editor
  • Migration defaults: current author → reporter, current editor → copy_editor, current viewer → reader
  • team page becomes accessible to admin and super_admin
  • system_settings becomes super_admin only
  • newsroom_settings becomes admin + super_admin

2. Panel Structure

3. Team and Auth Flow

  • Team invite dropdown shows admin, reporter, copy_editor, plus super_admin only when the acting user is a Super Admin
  • Setup/reset link flow remains the same
  • Login continues through /signin?redirect=/admin
  • loginId or email + password remains valid

4. Reporter Submission Data

  • Reporter forms must support media upload, location tagging, and source info
  • Minimum content additions: locationTag, sourceInfo, optional sourceConfidential, optional reporterNotes

5. Copy-Editor Quality Data

  • Track proofread completion, fact-check status, headline revision status, image optimization status, copy editor notes, and return-for-changes reason

6. Admin Operations Surfaces

  • Story Approval
  • Assignment Desk
  • Content Queue
  • Push Alerts
  • User Management
  • Newsroom Settings

Important API and Type Changes

  • Replace old admin role vocabulary with the four-role model
  • Add ready_for_approval and changes_requested to the shared workflow contract
  • Split settings permissions into system_settings and newsroom_settings
  • Restrict team-management APIs so only super_admin can manage super_admin
  • Extend content submission/update payloads with locationTag, sourceInfo, and copy-edit review metadata

Phase-by-Phase Build Order

Phase 1: Role Cleanup

Remove viewer from admin access, map author to reporter, map editor to copy editor, update labels, keep backward-compatible user migration.

Phase 2: Permission Enforcement

Rewrite page access and action permissions, move Team/User Management to admin + super_admin, keep System Settings super_admin only, add Newsroom Settings for admin.

Phase 3: Dashboard Split

Build distinct dashboards for Super Admin, Admin, Reporter, and Copy Editor.

Phase 4: Workflow Enforcement

Implement reporter submission, admin assignment and approval, copy-editor review, and push-alert handling.

Phase 5: Form and Queue Upgrades

Add reporter metadata, copy-editor review metadata, content queue, assignment desk, and push-alert panels.

Phase 6: Migration and QA

Migrate old roles, validate access per role, verify invite/setup/login flows, and close leftover old labels/routes.

Test Plan

Permissions

  • Super Admin can access system settings and revenue panels
  • Admin cannot access system settings or revenue panels
  • Admin can manage admin, reporter, and copy_editor
  • Admin cannot create, edit, or remove super_admin
  • Reporter cannot access review queue, team, analytics, or settings
  • Copy Editor cannot access team, push alerts, or system settings

Workflow and Auth

  • Reporter can create and submit only their own content
  • Copy Editor can edit assigned items and mark ready for approval
  • Copy Editor can return changes requested
  • Admin can approve, reject, schedule, publish, and push alert
  • Super Admin is not required for normal publishing flow
  • Generated setup links, login IDs, and password reset links all work

Assumptions and Defaults

  • Revenue & Ads Control is a dedicated Super-Admin-only surface, even if initially delivered as a placeholder
  • Newsroom Settings is separate from System Settings
  • Admin has full newsroom user management, but not platform-owner control
  • Reporter is submission-focused, not approval-focused
  • Copy Editor is quality-focused, not publishing-focused
  • reader remains the non-admin public-site role
